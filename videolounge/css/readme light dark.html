A) Introduction - top

CSS3 Form Pack features a large collection of form styles that are used regularly in websites. You get 66 Search Forms, 33 Login Forms, 12 Contact Forms, and one General Form. Further all these forms are available in Dark and Light coloured versions, so you get a total of 224 form styles! The login, contact and general forms also have focus and error styles defined in the css.

All these forms have been styled using CSS3, and no images except icons have been used. Also no javascript has been used except on older browsers like IE 8 to achieve some of the functionality that modern browsers can easily do. Whatever javascript has been used are present at the bottom of the demo html files. There is only one external javascript file called placeholder.js which is used to produce the placeholder effect in older browsers. It should be noted that all javascript has been written by utilizing the jQuery javascript library. All the various form styles can be easily used by just inserting some class names to your form. The process of using the forms has been described in the sections below.

B) Search Forms - top

There are two main types of Search Forms - a) with a Submit button, b) without a Submit button. The html code for the first variety is:

<form action="" class="search-form">
    <input type="text" name="search" class="search-input" placeholder="Search..." />
    <input class="search-btn" type="submit" value="Go" />       
</form>

Similarly the html code for the search form without a Submit button is:

<form action="" class="search-form">
    <input type="text" name="search" class="search-input" placeholder="Search..." />
</form>

The rest is handled by css styling. For that some classes have to be applied to the search form. You can already see that the form has the class search-form. This is essential for all the styles to take effect. We have two varieties of class names – those that change the shape of the form, and those that change the colour of the submit button. We will now discuss all these classes.

The search forms are either enclosed in a frame or they are open. If you want the first variety then add the class frame, and if you do not want a frame then add the class noframe. All the other class names that can be applied to the search forms are given below.

Note that the search forms by default have straight corners, you only need special classes if you want round corners. So if you want a search form that has a frame, has large rounded corners and the submit button is of light gray colour and is outside the search field, then you will apply the class names to the form as shown below: 

<form action="" class="search-form frame rbtn rlarge lgray">
    <input type="text" name="search" class="search-input" placeholder="Search..." />
    <input class="search-btn" type="submit" value="Go" />       
</form>

C) Login Forms - top

Login forms have been been broadly divided into two categories – a) labels on top of form fields, b) labels as placeholders inside form fields. The basic html structure for the login form where the labels are on top is: 

<form action="" class="login-form">
    <label for="username">Username</label>
    <input type="text" name="username" class="login-input" required />
    
    <label for="password">Password</label>
    <input type="password" name="password" class="login-input" required />
    
    <span class="forgot">Forgot <a href="#">Username</a> / <a href="#">Password</a></span>
    
    <input type="submit" class="login-btn" value="Login" />
</form>

The html structure for login forms with placeholders is:

<form action="" class="login-form">
    <input type="text" name="username" class="login-input" placeholder="Username" required />
    
    <input type="password" name="password" class="login-input" placeholder="Password" required />
    
    <span class="forgot">Forgot <a href="#">Username</a> / <a href="#">Password</a></span>
    
    <input type="submit" class="login-btn" value="Login" />
</form>

As you may have noticed the login forms must have the class login-form. To achieve further styling a few more class names are available which are described below.

D) Contact Forms - top

There are 6 available styles of contact forms and each style can have field labels either on top of the form fields or inside the fields as placeholders. The general html structure of a contact form with labels on top is: 

<form action="" class="contact-form">
    <label for="name">Name <span>(required)</span></label>
    <input type="text" name="name" class="form-input" required />

    <label for="email">Email <span>(required)</span></label>
    <input type="email" name="email" class="form-input" required />

    <label for="subject">Subject <span>(optional)</span></label>
    <input type="text" name="subject" class="form-input" />

    <label for="message">Message <span>(required)</span></label>
    <textarea name="message" class="form-input" required></textarea>

    <input class="form-btn" type="submit" value="Send Message" />       
</form>

Th html structure for a contact fom with labels as placeholders is:

<form action="" class="contact-form nolabel">
    <input type="text" name="name" class="form-input" placeholder="Name (required)" required />

    <input type="email" name="email" class="form-input" placeholder="Email (required)" required />

    <input type="text" name="subject" class="form-input" placeholder="Subject (optional)" />

    <textarea name="message" class="form-input" placeholder="Message (required)" required></textarea>

    <input class="form-btn" type="submit" value="Send Message" />       
</form>

The styles that are available can be implemented by using the following class names on the form:

  • frame
  • frame tbar
  • msgtop
  • form3d
  • envelope

The simple form style that is shown at the beginning in the demo does not need any special class. Also the form styles that are achieved by using the class names msgtop and envelope have a slightly different html structure which you can study in the demo.

Please note that the javascript file placeholder.js, that provides the placeholder text effect in older browsers, is slightly modified here to also take care of textareas. So don't mix up the placeholder.js file from the Contact From folder with those present in other folders.

In the general form, styles for the following form elements have been provided:

F) Instructions for IE - top

IE9 supports more CSS3 features than its previous versions. So very few tricks/hacks are required for this browser. It however does not support CSS3 gradients. To make the forms look good in IE 8 and lower versions CSS3 PIE and jQuery based fixes have been used. CSS3 PIE is also used to generate gradients in IE9. The scripts that have been included at the bottom of the html files, add a class to the <html> element called ie when IE8 or lower versions are detected. This class allows us to target these bowsers specifically in the css. Only for the styling of the select box in the general form we have to apply a separate class called ie9 to target IE9. Besides CSS3 PIE we also have to include Dean Edwards' IE9.js in the <head> like this: 

<!--[if lt IE 9]>
    <script src="http://ie7-js.googlecode.com/svn/version/2.1(beta4)/IE9.js"></script>
<![endif]-->

IE9.js makes IE < 9 more standards compliant and enables support for many modern CSS3 selectors. Please note that generally the url path of a htc file (CSS3 PIE) which is declared in the css file, is made relative to the html file, and not relative to the css file from where it is referenced. Dean Edwards' IE9.js corrects this non-standard behaviour in IE, and hence you can declare the htc file path relative to the css file, just like background images. However IE9 still requires the htc file path to be relative to the html file. Since in the demo all files, whether html or css are present together, this does not pose a problem. So whatever path is relative to the css file is also relative to the html. But in your project you will most likely have the css file present in a separate folder, and also obviously there cannot be two declarations for the path of the htc file. So in that case the best way would be to keep two copies of the CSS3 PIE htc file – one inside the css folder and one at the same level as the html file. Therefore you can declare the htc file in the css as behavior: url(PIE.htc); and this path will be relative to the css file for IE < 9 and relative to the html file for IE9, and hence all versions of IE will be able to load the htc file.

IE requires that HTC behaviors are served up with a content-type header of "text/x-component", otherwise it will simply ignore the behavior, IE9 being especially strict in this regard. Many web servers are preconfigured to serve the correct content-type, but others are not. If you have problems with the PIE behavior not being applied, check your server configuration and if possible update it to use the correct content-type. For Apache, you can do this in a .htaccess file: 

AddType text/x-component .htc

You will also see in the demo that HTML5 Shiv is used. If you are using HTML5 in your pages then you may already have this included, but this file is not required for the styling of the forms, but has only been used in the demo because the demo uses modern HTML5 tags.

G) Form validation styles - top

Error styles have been provided for the forms when they fail validation. HTML5 now provides client side validation without the need of any javascript based method. In order to style invalid form fields a new pseudo class name is available, and it is called :invalid. However there is a problem with this pseudo class, and that is that the browser applies this class to form fields on page load. So if for example you have a login form where both the username and password fields are required, then when the page loads the browser applies the class :invalid to these fields because these fields are empty (naturally) on page load. So if the :invalid pseudo class has been styled to produce a red glow then users will see a red glow when the page loads. This is against good user experience and will mislead users. Only Firefox provides a solution for this in the form of a new pseudo class called :-moz-ui-invalid. This class is activated only when the browser validates the form on submit. Therefore this class provides a better user experience and behaves just like standard javascript based solutions. However no such class exists in other browsers. Therefore a class called invalid is provided in the css which you can apply using a javascript based validation method. To illustrate this the demo files use the new HTML5 event called oninvalid (which is fired when form validation fails) to apply the class invalid to the form fields. Please note that this HTML5 event works in only Firefox, Chrome and Opera (although this event has not been utilized in Firefox in the demo).
